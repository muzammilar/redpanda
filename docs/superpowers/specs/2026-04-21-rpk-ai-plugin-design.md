# rpk ai: rpai as a managed rpk plugin

Date: 2026-04-21
Status: design approved, implementation pending

## Goal

Make `rpai` (the Redpanda AI Gateway CLI that lives in `cloudv2/apps/rpai`) invokable as
`rpk ai <subcommand>` by adding a managed rpk plugin. Users should be able to:

- Run `rpk ai install` / `rpk ai upgrade` / `rpk ai uninstall` to manage the plugin binary
  the same way they do for Redpanda Connect.
- Run `rpk ai <subcommand>` against a cluster selected via their current rpk cloud
  profile — no separate `rpai auth login`, no second credentials file.
- Keep using standalone `rpai` (installed via Homebrew) without behavior changes.

## Non-goals

- Changing how standalone `rpai` works (its profile system, OAuth device flow, config files).
- Shipping a FIPS-specific manifest. Revisit when/if aigw ships FIPS artifacts.
- Dropping rpai's Homebrew distribution. The managed-plugin install path and brew install
  path coexist.
- Supporting self-hosted aigw deployments that aren't reachable through the Redpanda Cloud
  public API. Users with custom endpoints keep using standalone rpai, or pass
  `--rpai-endpoint`.

## Context

rpai today ships as a standalone Go CLI distributed via Homebrew and Cloudsmith tarballs
(`dl.redpanda.com/public/rpk-plugins/raw/names/rpai-<os>-<arch>/...`) for six platforms
(darwin/linux/windows × amd64/arm64). It implements Redpanda AI Gateway management
(LLM providers, MCP servers, OAuth providers, models, etc.) over the aigw Connect-RPC API.

RFC 003 (`cloudv2/adp/rfcs/003-rpai-cli.md`) already anticipated rpk integration as
"Phase 2": the Homebrew formula would drop a `.rpk.ac-ai` symlink so rpk could discover
rpai on PATH, with flag-rename logic and a `--help-autocomplete` stub baked into rpai to
handle plugin mode. That design keeps rpk-side code out of it entirely.

We are instead choosing the "managed plugin" model, mirroring `rpk connect`. Reasons:

- Discoverability: `rpk ai` appears in `rpk --help` regardless of whether the plugin is
  installed; `rpk ai install` is the blessed install command.
- Auth UX: `rpk cloud login` should be sufficient to run `rpk ai …`, without a separate
  `rpai auth login`. This needs rpk-side code to refresh the cloud token and pass it to
  rpai.
- Consistency with connect/byoc precedents already in rpk.

## Architecture overview

Two PRs land together (coordination-friendly but independent):

### redpanda PR

New package `src/go/rpk/pkg/cli/ai/` modeled closely on `src/go/rpk/pkg/cli/connect/`:

```
src/go/rpk/pkg/cli/ai/
├── ai.go         // init() registers managed plugin; exports NewCommand
├── client.go     // manifest HTTP client
├── install.go    // rpk ai install
├── uninstall.go  // rpk ai uninstall
├── upgrade.go    // rpk ai upgrade
├── hook.go       // token injection + endpoint derivation
└── BUILD         // auto-generated by Gazelle via `make bazel`
```

The `ai.NewCommand(fs, p, execFn)` return value is wired into `src/go/rpk/pkg/cli/root.go`
alongside the existing `connect.NewCommand(...)` invocation.

### cloudv2 PR

Two changes under `cloudv2/apps/rpai`:

1. Extend the release pipeline (`rpai-release.yml`, and optionally `rpai-snapshot.yml`) to
   publish a JSON manifest at `https://rpk-plugins.redpanda.com/rpai/manifest.json`. Schema
   identical to `connect`'s manifest.
2. Tighten `detectPluginMode()` so that when rpai is invoked as `.rpk.managed-rpai`, it
   suppresses its `profile` and `auth login`/`auth logout` subtrees, skips reading
   `~/.rpai/config` + `~/.rpai/credentials`, and takes its token/endpoint exclusively from
   env vars (`RPAI_TOKEN`, `RPAI_ENDPOINT`) or explicit overrides (`--rpai-endpoint`).

### Runtime flow

User runs `rpk ai llm list`:

```
rpk
 └─ cobra routes to the "ai" node (registered via plugin.RegisterManaged)
    └─ managed hook (hook.go):
         - locate/auto-install .rpk.managed-rpai
         - load active rpk cloud profile; refresh OAuth token
         - resolve aigw endpoint for active cluster (via publicapi)
         - export RPAI_TOKEN + RPAI_ENDPOINT
         - strip rpk-global flag collisions (re-use cobraext.StripFlagset)
    └─ exec ~/.local/bin/.rpk.managed-rpai llm list
```

### Install flow

User runs `rpk ai install [--rpai-version X.Y.Z]`:

```
rpk
 └─ rpk.ai.installCommand:
      - GET https://rpk-plugins.redpanda.com/rpai/manifest.json
        (RPK_PLUGIN_REPOSITORY env overrides base URL, matching connect)
      - pick latest archive (or matching version)
      - plugin.Download(...) with expected SHA256
      - plugin.WriteBinary(fs, "rpai", DefaultBinPath, bin, false, true)
        → writes ~/.local/bin/.rpk.managed-rpai
```

## Component design

### Managed plugin registration (`ai.go`)

```go
func init() {
    plugin.RegisterManaged("rpai", []string{"ai"}, func(cmd *cobra.Command, fs afero.Fs, p *config.Params) *cobra.Command {
        run := cmd.Run
        cmd.Run = func(cmd *cobra.Command, args []string) {
            pluginArgs, err := applyHook(fs, p, cmd, args)
            out.MaybeDie(err, "unable to prepare rpk ai invocation: %v", err)
            run(cmd, pluginArgs)
        }
        return cmd
    })
}
```

`NewCommand` owns the top-level `rpk ai` cobra command with `DisableFlagParsing: true`,
auto-installs on first subcommand use (pattern from `connect.NewCommand`), and attaches
`install`/`uninstall`/`upgrade` subcommands.

### Manifest client (`client.go`)

Near-copy of `connect/client.go`. Same structs (`artifact { path, sha256 }`,
`archive { version, is_latest, artifacts }`, `manifest { archives }`). Different path:

```go
const pluginBaseURL = "https://rpk-plugins.redpanda.com"
// manifest path: /rpai/manifest.json
```

Honors `RPK_PLUGIN_REPOSITORY` env override so dev/CI can point at a staging manifest.

### Install commands (`install.go`, `uninstall.go`, `upgrade.go`)

Copy the connect analogues verbatim with these edits:

- Flag name: `--rpai-version` (vs connect's `--connect-version`).
- Plugin key: `"rpai"` so `plugin.WriteBinary` produces `.rpk.managed-rpai`.
- Error strings and help text refer to "Redpanda AI CLI" / "rpai".

Version validation reuses the semver regex from `connect.validateVersion`.

### Hook (`hook.go`)

The hook runs before exec. Ordered responsibilities:

1. **Resolve binary.** Find `.rpk.managed-rpai` on `plugin.UserPaths()`. If absent and the
   invocation is a subcommand (not `--help`), auto-install (identical branch to
   `connect.NewCommand`).
2. **Load rpk cloud session.** Open the active profile from `*config.Params`. If no cloud
   auth section exists, skip to step 5 and let rpai in plugin mode print a clear
   "run `rpk cloud login` first" error.
3. **Refresh token.** Use the same refresh path `rpk cloud ...` commands use (e.g.,
   `oauth.LoadFlow`). On failure, propagate the rpk-style error.
4. **Derive aigw endpoint.** Call the public API for the active cluster and read the aigw
   URL. See "Open items" — the exact field/mechanism needs confirmation.
5. **Env + flag handling.** In each case, skip the override if the variable is already
   set in the environment — an explicit `RPAI_TOKEN`/`RPAI_ENDPOINT` from the caller wins.
   - If step 3 succeeded and `RPAI_TOKEN` is unset: `os.Setenv("RPAI_TOKEN", refreshedToken)`.
   - If step 4 succeeded and `RPAI_ENDPOINT` is unset: `os.Setenv("RPAI_ENDPOINT", endpoint)`.
   - Run `cobraext.StripFlagset` to split rpk-global flags (`-v`, `--verbose`, `--config`,
     `--profile`) from plugin args, matching `parseConnectFlags`.
6. Return the filtered args to the caller, which exec's the plugin binary.

Failure-mode rules:

- Missing cloud session ⇒ don't fail in the hook; let rpai print its plugin-mode error so
  the user sees a consistent rpai-style message.
- Expired/non-refreshable token ⇒ fail in the hook with an rpk-style error pointing at
  `rpk cloud login`.
- Unknown cluster / missing aigw URL ⇒ fail in the hook with a message telling the user to
  switch profile or use standalone rpai.

### rpai plugin-mode behavior (cloudv2 side)

Gated on the argv0 of the invoked binary being `.rpk.managed-rpai` (symlink-resistant: use
`os.Executable()` and read the final filename). When detected:

- `cmd/rpai/main.go` → `internal/cmd.Execute()` → root command construction suppresses the
  `profile` and `auth login`/`auth logout` cobra subtrees (leave `auth status` and
  `auth token` as read-only diagnostics if useful; revisit during implementation).
- Config loading bypasses `~/.rpai/config` and `~/.rpai/credentials` entirely.
- Token/endpoint come from `RPAI_TOKEN` / `RPAI_ENDPOINT` / `--rpai-endpoint` only.
- `--config`, `--profile`, `--verbose` global flags rename to `--rpai-config`,
  `--rpai-profile`, `--rpai-verbose` (already scaffolded in rpai per RFC 003; this change
  turns the rename on in plugin mode).
- `--help-autocomplete` emits the full cobra command tree as JSON so rpk's autocomplete
  wiring works.

### Manifest publishing (cloudv2 release pipeline)

After tarballs upload to Cloudsmith and the HTTP-200 verification gate passes, a new step
merges the new release into the manifest JSON and publishes it.

Mechanism: same bucket/CDN pattern `rpk-plugins.redpanda.com/connect/manifest.json` already
uses. Concrete steps (confirm during implementation):

- Fetch current manifest, clear the `is_latest` flag on the previous latest archive, append
  the new archive with `is_latest: true`.
- Compute SHA256 from Cloudsmith-reported checksums.
- Publish back to origin with the same credentials connect's release job uses.

Snapshot builds optionally publish a parallel manifest (e.g., `/rpai-snapshot/manifest.json`)
for dev override via `RPK_PLUGIN_REPOSITORY`. Decide during implementation; not required.

## Testing

### Unit (redpanda repo)

- `client_test.go`: adapt `connect/client_test.go`. Manifest parsing, missing os/arch,
  missing version, respecting `RPK_PLUGIN_REPOSITORY`.
- `install_test.go`: adapt `connect/install_test.go`. Happy path, `--force`, already-installed,
  version validation.
- `hook_test.go`: new. Matrix across no-session / expired-token / missing-cluster / happy.
  Mock the cloud session loader and public API client; assert env vars set, flags stripped,
  errors surfaced.

### Integration

- Manifest download + binary write through an `httptest.NewServer` producing a valid
  SHA256-checksummed tarball, matching the connect test fixtures.

### Manual smoke (release-readiness)

1. `rpk cloud login` against dev.
2. `rpk ai install` — binary lands in `~/.local/bin/.rpk.managed-rpai`.
3. `rpk ai llm list` — exec's, talks to dev aigw, returns rows.
4. `rpai llm list` standalone — unchanged behavior, uses `~/.rpai/credentials`.

### Regression

- rpai's existing cloudv2 CI tests confirm plugin-mode detection doesn't trigger during
  regular `rpai` invocations.
- rpk's existing plugin tests still pass (no regressions in `plugin.RegisterManaged`
  semantics).

## Open items (resolved at implementation time)

- **aigw endpoint resolution.** Mechanism for going cluster ID → aigw URL. Preferred:
  new or existing field on the cluster proto returned by `publicapi`. Fallback: derive by
  convention (e.g., `aigw.<cluster-dns-name>`) — but confirm before coding. This is the
  single biggest schedule risk; may need a companion cloud server PR.
- **rpk-plugins.redpanda.com write credentials.** Which job/role publishes connect's
  manifest today? Reuse that pattern exactly. Name the owner during implementation.
- **`rpai auth status` / `rpai auth token` in plugin mode.** Keep visible (read-only
  diagnostics) or hide? Lean toward keep visible; finalize when wiring
  `detectPluginMode()`.
- **Snapshot manifest.** Worth publishing `/rpai-snapshot/manifest.json` for dev overrides?
  Optional, decide during implementation.
- **`rpai auth` UX parity for the managed path.** Standalone rpai users run `rpai auth
  login` once and forget it; managed users rely on `rpk cloud login`. Document this
  clearly in both `rpk ai --help` and the rpai README.
